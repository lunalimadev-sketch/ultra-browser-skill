# Tutorial: Otimizando o Startup do OpenClaw

Este documento registra as configurações aplicadas para resolver o problema de longos delays (chegando a minutos) durante o startup do gateway OpenClaw no Windows.

## 1. Desabilitar Bonjour / mDNS

**Sintoma:** O gateway pausa por vários segundos/minutos. Nos logs (`openclaw logs --follow`), aparecem mensagens como `[bonjour] watchdog detected non-announced service` e `restarting advertiser (service stuck in announcing)`.
**Causa:** A biblioteca `@homebridge/ciao` (usada para local discovery) trava quando a interface de rede do Windows não suporta nativamente multicast ou possui configurações restritas.
**Solução:** Se o acesso ao gateway não depende de discovery em rede local (ex: usado direto via Telegram ou localhost), desabilite o serviço.

**Como aplicar:**
Abra o arquivo `~/.openclaw/.env` e adicione a seguinte variável:
```env
OPENCLAW_DISABLE_BONJOUR=1
```

## 2. Otimizações no `openclaw.json` (Reduzir overhead de rede e plugins)

**Sintoma:** Demora excessiva no hook `[gateway/model-pricing]` emitindo erros como `TimeoutError: The operation was aborted due to timeout` ou demoras não explicáveis após o carregamento dos hooks.
**Causa:** Verificações de telemetria, busca de updates no arranque e plugins ativados sem necessidade que tentam se inicializar e comunicam-se com a internet sincronamente durante o boot.

**Solução:**
Abra o arquivo `~/.openclaw/openclaw.json` e modifique as seguintes raízes:

1. **Desativar Checagem de Update no Startup:**
```json
{
  "update": {
    "checkOnStart": false
  }
}
```
*(Nota: dependendo da versão, opções como `telemetry.enabled` podem causar `Unrecognized key`, então evite-a se o `openclaw doctor` reclamar).*

2. **Desativar Plugins Inativos:**
Ainda no `openclaw.json`, desative plugins não rodando ativamente para poupar tempo de módulo (ex: se usar API Keys normais para os models e não precisar de autenticação pelo gateway Vercel):
```json
{
  "plugins": {
    "entries": {
      "vercel-ai-gateway": {
        "enabled": false
      }
    }
  }
}
```

## 3. Gestão Correta de Reinicialização (Crash na porta UDP/TCP)

**Sintoma:** Uso de `openclaw gateway restart` acusa `listen EADDRINUSE: address already in use 127.0.0.1:18789`.
**Causa:** O Windows Service Scheduler ou o Node demoram para liberar a porta após um comando direto de restart, causando sobreposição de processos falhos.

**Solução:**
Ao invés de usar `restart`, mate as instâncias explicitamente e aguarde liberação (sleep/timeout) antes de religar:
```powershell
openclaw gateway stop
Start-Sleep -Seconds 3
openclaw gateway start
```

### Resultados da Otimização
Estas três etapas combinadas transformaram uma inicialização que antes gerava mais de **4 minutos de bloqueio e timeouts**, para uma conclusão limpa de startup demorando apenas cerca de **~35 segundos** na nossa máquina.
